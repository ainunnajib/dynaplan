## Architecture
- Backend: FastAPI, 1530 tests (latest run: 1530 passed), file-backed SQLite with NullPool for tests (+ optional PostgreSQL test-container path)
- Engine: Pure Python formula engine in production now includes F051 time functions, F052 lookup/cross-module functions, F053 text/conversion functions, and F054 advanced aggregation/statistical functions (SUMIF/COUNTIF/AVERAGEIF, MEDIAN/STDEV/VARIANCE, PERCENTILE/LARGE/SMALL, GROWTH) with matching Rust formula-module implementation (`engine/dynaplan-core/src/formula/functions.rs`) pending Rust toolchain validation
- Quotas: Workspace-level quota controls (max models, max cells/model, max dimensions/model, storage MB) with API usage reporting + enforced create/write guards
- Database infra: Alembic runtime + revision chain in `backend/alembic`, asyncpg pool controls, optional read-replica session dependency
- Execution runtime foundation: async background executor (`job_executor`), scheduler wrapper (`job_scheduler`), active job registry (`job_registry`), and CloudWorks connector SDK (`app/connectors`) with run execution endpoint
- Frontend: Next.js 15 + TypeScript + Tailwind + Recharts, TS compiles clean
- Cell storage: sparse table with sorted pipe-separated dimension_key
- API keys: hashed with SHA-256, scoped (read:models, write:cells, admin, etc.)
- Import/export: CSV and Excel via openpyxl
- Actions: chained processes with step ordering, async run
- Collaboration: WebSocket presence + cursor + cell broadcasts
- RBAC: workspace roles, model permissions, dimension member access
- DCA: selective access rules + driver line items for cell-level read/write control
- SSO: SAML 2.0 + OIDC provider config
- Workflows: stages, tasks, approvals, gate checks
- ALM: dev/test/prod environments, revision tags with auto snapshots, deep structural diff/change detection, conflict-aware promotion engine with additive/replace/manual strategies, execution-time rollback restore, and promotion records
- CloudWorks: scheduled imports/exports with retry/status tracking plus connector-driven execution (`/runs/{run_id}/execute`)
- Pipelines: no-code data orchestration with step-level logging
- SCIM: automated user/group provisioning with enterprise IdP integration
- Engine profiles: Classic and Polaris-like modes for large model optimization
- Backpressure: `pytest -m backpressure` marker added with dedicated CI gate
